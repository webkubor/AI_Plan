#### 应用层：AI 落地形态

- **Agent（智能代理）**：基于 LLM 的自主决策系统，能感知环境、制定计划并执行任务，具备 “思考 - 行动 - 反馈” 的闭环能力。**核心特征**：无需人工干预即可完成复杂任务，如自动生成代码、撰写报告、处理客服咨询。
- **Skill（技能）**：Agent 可调用的模块化能力单元，是将通用 LLM 适配特定场景的 “插件”。例如，代码生成、数据可视化、API 调用等都属于 Skill，让 AI 能精准解决细分领域问题。
- **Workflow（工作流）**：将多个 Skill 或 Agent 按业务逻辑串联起来的自动化流程，实现端到端的任务自动化。例如，“需求文档生成→代码开发→单元测试→部署上线” 的全流程自动化。



#### **MCP（Model Context Protocol）：模型层的 “交互协议”**

- **核心定义**：MCP 是由 OpenAI 等厂商提出的**模型间通信协议**，本质是一套标准化的格式和规则，让不同的 AI 模型（如 LLM、CV 模型、语音模型）能够互相理解、传递信息、协同完成任务。
- **层级归属**：属于**AI 基础层中的 “核心模型层”**，是连接不同类型模型的 “中间件”，解决了多模型协作时的 “语言不通” 问题。
- **通俗类比**：如果把各个 AI 模型比作不同国家的专家，MCP 就是 “通用翻译官”，让擅长代码的 Codex、擅长图像的 GPT-4V、擅长语音的 Whisper 能无障碍沟通，共同完成复杂任务。
- **前端场景应用**：比如你要实现 “用户上传支付界面截图→AI 分析 UI 问题→生成优化代码” 的流程，MCP 可以让 CV 模型（分析截图）将结果标准化传递给 LLM（生成代码），无需你手动处理数据格式转换。

#### 2. **Agent（智能代理）：应用层的 “自主执行者”**

- **核心定义**：Agent 是基于 AI 模型（通常是 LLM）构建的**自主决策执行系统**，具备 “感知 - 规划 - 行动 - 反馈” 的完整闭环能力，无需人工干预即可独立完成复杂任务。

- **层级归属**：属于**AI 应用层**，是基础层模型（LLM、MCP）的 “落地载体”，将模型能力转化为可直接解决业务问题的工具。

- **关键特征（区别于单纯的 LLM 调用）**：

  - 自主决策：能根据任务目标拆解步骤，比如 “生成收银台组件” 会拆解为 “确定需求→设计 UI→编写代码→生成测试”；
  - 工具调用：可以主动调用外部工具（如代码编译器、API、数据库），比如遇到支付金额计算问题时，自动调用计算器工具或后端接口；
  - 记忆能力：能记住历史交互信息，比如记住用户之前的支付偏好，优化后续推荐。

- **与你现有能力的关联**：你现在调用 LLM API 实现对话，只是 “被动响应”；而 Agent 是 “主动完成任务”，比如你给 Agent 下达 “优化收银台移动端支付流程” 的指令，它会自动拆解任务、调用工具、输出最终优化方案和代码。

  **领域典型 Agent 工具**：

  **Cursor Agent**：Cursor 内置的 Agent 功能，可自动完成 “需求拆解→代码编写→调试优化” 的全流程；

#### 3. **Vibe Coding（氛围编码）：工作流层面的 “AI 协作模式”**

- **核心定义**：这是一种**人机协同的编码方式**，指开发者在终端或编辑器中，与 AI 模型（如 Codex、Gemini）进行实时、自然的交互，像和同事协作一样边沟通边编码，而非单纯让 AI “一次性生成完整代码”。
- **层级归属**：不属于 AI 技术层级，而是**AI 赋能开发的 “工作流模式”**，是你已掌握的 “工具使用能力” 的高阶形态。
- **核心特点**：
  - 实时交互：开发者写几行代码，AI 就自动补全后续逻辑，或根据开发者的自然语言指令（如 “给这个按钮加个加载状态”）实时修改代码；
  - 创意共创：开发者提供思路和框架，AI 补充细节和实现方案，比如你设计收银台的支付逻辑，AI 帮你完善边界条件和错误处理；
  - 沉浸式体验：无需切换工具，在编码环境中直接完成 “思考 - 指令 - 实现” 的闭环，大幅提升开发效率和流畅度。

Agent 通过 **MCP（对讲机）**，协调多个 “专业工具” 协同工作：

- 调用 **Midjourney（创意设计师）**：生成 ins 风青提柠檬茶的外观效果图，确认颜值符合要求；
- 调用 **CV 模型（颜值检测仪）**：实时监控制作过程，确保配料摆放、杯型装饰和效果图一致；
- 调用 **Codex（奶茶制作机）**：按照配方精准制作青提柠檬茶，控制糖度和冰量；
- 调用 **Skill（打包技能）**：完成打包，并生成包含营养成分的小票。

### 核心逻辑公式（一句话总结）

**用户 Prompt（需求）→ Agent（决策者）→ 拆解 Task（任务）→ 按 Workflow（流程）→ 通过 MCP（协议）→ 调用 Skills（工具能力）→ 完成目标**



#### cursor是什么，Agent 吗？

**集成了 Agent 能力的 “执行终端”**，但 Agent 本身不是编辑器，而是编辑器背后的 “智能决策大脑”；且 “输入 Prompt 就从写代码到上线”，本质是编辑器里的 Agent 串联了多工具的 Skills，再通过 Workflow 完成全流程，而非编辑器单独做到。



**大模型是 Agent 的 “大脑内核”，但 Agent 不是大模型本身**。

团队里的技术负责人是懂业务、会拆解任务、协调资源、把控流程，能把一个模糊目标落地成可上线产品

**大模型（GPT-4o/Gemini）**：就像顶级程序员，拥有超强的代码生成、语言理解能力，但本质是 “被动执行指令”，不会主动拆解复杂任务、调用外部工具

技术架构上是

```js
Agent = 大模型（LLM）+ 任务拆解模块 + 工具调用模块 + 记忆模块 + 反馈优化模块
```
